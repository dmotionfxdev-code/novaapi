## 0. Pre-Acceptance Incident (already resolved, included for continuity)

Before this test could run at all, `POST /api/v1/auth/token` and `POST /api/v1/tenants` were both returning `500`. Root-caused via direct connection to the live Postgres (in the order requested): `alembic_version` did not exist, zero of the 10 expected logical schemas existed, zero tables existed — migrations had never been applied, because Render's free tier rejects `preDeployCommand` outright and that step had never been run manually. Fixed by running `alembic upgrade head` directly against the live database (all 20 revisions, `0000_baseline` → `0019_security_hardening`, applied cleanly). This report's test run is the first real, full exercise of the platform *after* that fix — itself part of the acceptance evidence that the fix holds.

## 15. Remote Sensing (GEE) — credentials present but malformed

```
POST /api/v1/dataset-sources {"provider": "GOOGLE_EARTH_ENGINE", ...} → 201
POST /api/v1/acquisition-jobs {"remote_sensing_source": "SENTINEL_2", ...} → 201, status: "SCHEDULED"
POST /api/v1/acquisition-jobs/{id}/actions/execute → 200, status: "FAILED"
error: "Unable to load PEM file. See https://cryptography.io/en/latest/faq/#why-can-t-i-import-my-pem-file
        for more details. InvalidData(InvalidByte(0, 92))"
```
**This is a genuinely interesting, real finding, traced to source rather than assumed.** `infrastructure/gee_connector.py:213` has an explicit unconfigured-credentials guard: `if not self._service_account_email or not self._service_account_private_key:` → returns a clean `"Google Earth Engine is not configured"` message. That guard was **not** triggered — meaning `GEE_SERVICE_ACCOUNT_EMAIL`/`GEE_SERVICE_ACCOUNT_PRIVATE_KEY` currently hold **non-empty values** on this Render deployment, but the private key fails to parse as a PEM certificate (`InvalidByte(0, 92)` — byte 92 is `\`, at position 0, consistent with a literal `\n` escape-sequence pair being pasted into Render's dashboard instead of the key's real embedded newlines, which is exactly the failure mode `.env.production.example`'s own instructions warn about: *"the FULL contents of the downloaded service-account JSON key file, minified to a single line"*).

This is **not a code bug** — the honest-failure guard exists and is correctly written; it just isn't the guard that fired here, because these fields aren't actually empty. **Action needed, if Remote Sensing is required**: re-enter `GEE_SERVICE_ACCOUNT_PRIVATE_KEY` in Render's dashboard making sure literal newlines in the key are preserved as `\n` two-character escape sequences in the JSON *value itself* (not doubly-escaped, and not raw newlines pasted into a single-line dashboard field), or clear both fields entirely to fall back to the clean "not configured" behavior demonstrated everywhere else in this platform. If Remote Sensing isn't needed yet, no action is required — every other capability tested above works correctly regardless.

### 15a. Update — credential fixed, then a real bug found and fixed

After this report's original run, the private key was corrected (real newlines, re-verified via a
byte-accurate extraction rather than manual retyping) and re-tested live. Authentication then
succeeded, surfacing a **second, genuine issue**: any AOI larger than a small test square made the
raw raster download exceed Earth Engine's fixed 48MB synchronous request-size limit, failing the
**entire acquisition** — even though the real, useful output (`band_statistics`, computed via
`reduceRegion` moments earlier) had already succeeded. Traced precisely: the raw pixel bytes are
never consumed by anything downstream in this platform (no raster pipeline exists in Analysis,
Prediction, Validation, or the frontend — confirmed by direct inspection of every consumer of
`FetchResult.content`). Classified as a **Bug** (existing, shipped capability not actually working
for realistic inputs) and fixed: the raster download is now a best-effort, optional artifact — an
Earth-Engine-size-limit failure is caught specifically (not swallowed broadly) and the acquisition
completes successfully using the already-computed statistics, with the omission honestly recorded
in the job's provenance rather than fabricating placeholder content. Verified with 4 new unit tests
(`tests/unit/test_gee_connector.py`, including a test proving a genuine pre-download Earth Engine
failure still fails the job — the fix's error handling is not overly broad) and 2 new real-network
integration tests (`tests/integration/test_gee_connectivity.py`). Full detail:
`RELEASE_CANDIDATE_CHECKLIST.md` §6.

## Operational Note: one transient network error, not a deployment defect

Partway through this test run (after ~91 successful sequential requests, roughly 15 minutes into a long-lived HTTP connection), a single request failed with `SSL: SSLV3_ALERT_BAD_RECORD_MAC` — a TLS-layer read error, not an application error (no HTTP status was even returned). Recovered by re-authenticating with a fresh connection and continuing; steps 1-14 above were entirely unaffected (they'd already completed and their state is durable in Postgres) and step 15 was re-run and completed cleanly on the fresh connection. This looks like an ordinary long-lived-connection hiccup (either this sandbox's outbound network or an intermediate proxy) rather than anything wrong with the Render deployment — every retried/re-fetched call succeeded immediately after. Not something to fix; noted for completeness since the task asked for a fully evidenced report.

## Verdict

**The Render deployment is ready for frontend integration.**

Every real platform capability this task asked to verify — authentication (including the Sprint D revocation/rate-limiting hardening, already live-verified in an earlier pass this session), the full FIRAS Assessment→Analysis workflow, real Data Acquisition (JSON and Shapefile), real spatial Risk Layer generation and GeoJSON export suitable for direct map rendering, real Prediction (MLR with genuine statistical output), real Validation, real Report generation, real Notification triggering, and real Dashboard aggregation — worked correctly end-to-end over the live public URL, using only the same HTTP surface a real frontend client would use, with no shortcuts.

Two items are **not** deployment blockers but are worth carrying into frontend planning:
1. **No Activity Log endpoint exists** — if the frontend's design assumes one, that's new backend scope to plan for, not something broken in this deployment.
2. **Remote Sensing (GEE) needs its private key re-entered correctly** in Render's dashboard if that capability is needed on day one — every other Data Acquisition path (Local Upload, USGS/NASA/Copernicus-when-configured) is unaffected and already proven working (steps 4-5 above).

Also worth carrying forward operationally (from the earlier deployment audit, unrelated to this test): the free-tier web service sleeps after 15 minutes idle (30-60s cold start on the next request), and the free-tier Postgres auto-deletes after 30 days — both fine for continued frontend-integration work, both worth resolving (paid tiers) before any real production traffic.
